## 1. 文档目的

本文档用于明确手机终端在整个比赛项目中的职责、功能边界、通信方式、数据结构、安全要求、开发顺序和验收标准。

手机终端不是单纯的遥控器，也不是全部系统逻辑的唯一执行中心。其核心定位为：

> **老人个人状态数据采集入口、家庭物联网控制入口、告警与设备状态总览入口，以及 ESP32-C5、RDK X5 和智能家居设备的用户侧管理终端。**

手机端应与 ESP32-C5、RDK X5、智能家居设备和云端服务共同构成完整闭环：

```text
老人 / 家属 / 社区人员
        ↓
Android 手机终端
        ↓
ESP32-C5 本地交互与设备接入终端
        ↓
RDK X5 视觉、导航、运动和边缘计算节点
        ↓
智能灯、插座、空调、窗帘及其他家电
```

## 2. 项目整体定位

本项目面向居家养老场景，构建一个具备以下能力的 AIoT 系统：

1. 老人日常行为和状态数据采集；
2. 环境状态感知；
3. 机器人巡检和视觉复核；
4. 智能家居设备控制；
5. 异常事件分级判断；
6. 手机、ESP32 和 RDK 多端协同；
7. 本地与云端双路径运行；
8. 告警、执行结果和设备状态闭环反馈。

手机终端主要负责：

```text
用户身份与老人档案
行为数据采集
设备列表与状态查看
控制命令生成
告警接收与人工确认
场景配置
连接诊断
数据上传与同步
```

ESP32-C5 主要负责：

```text
本地屏幕与触摸
单个外接传感器管理
Wi-Fi 和手机本地接入
本地安全仲裁
RDK 指令转发
AI 链路自检
紧急停止
```

RDK X5 主要负责：

```text
视觉识别
人体姿态和跌倒疑似判断
导航与避障
小车底盘运动
复杂 AI 计算
云端通信代理
智能家居协议适配
```

## 3. 现有客户端基础能力

当前 Android 客户端已经具备一定基础，包括：

- 每日签到；
- 睡眠和起床异常检查；
- 步数统计；
- 应用使用状态统计；
- 紧急联系人管理；
- 异常确认；
- 短信和邮件告警；
- 后台定时任务；
- HTTP 数据上报；
- 守护对象管理；
- 告警历史记录；
- 本地配置和部分加密存储。

这些功能可继续保留，并作为老人状态辅助判断的数据源。

当前客户端尚未形成完整物联网控制能力，后续需要增加：

- ESP32 设备发现和绑定；
- ESP32 本地连接；
- RDK 状态显示；
- 智能家居设备注册；
- 命令发送和 ACK 跟踪；
- 设备状态同步；
- 场景调度；
- 本地离线控制；
- 统一诊断页面。

## 4. 手机终端总体软件架构

建议新增独立的 `iot` 功能模块，不将物联网功能直接堆入现有 `MainActivity` 或现有守护页面。

推荐结构：

```text

├── core/
│   ├── network/
│   ├── storage/
│   ├── security/
│   ├── logging/
│   └── common/
│
├── elder_profile/
│   ├── model/
│   ├── repository/
│   ├── service/
│   └── ui/
│
├── health_observation/
│   ├── activity/
│   ├── checkin/
│   ├── routine/
│   ├── risk/
│   └── ui/
│
├── iot/
│   ├── connection/
│   ├── discovery/
│   ├── protocol/
│   ├── device/
│   ├── command/
│   ├── scene/
│   ├── diagnostics/
│   ├── repository/
│   └── ui/
│
├── alert/
│   ├── model/
│   ├── service/
│   ├── notification/
│   └── ui/
│

```

```text
UI
  ↓

  ↓

  ↓

  ↓
```

UI 不应直接处理网络协议，也不应直接保存底层设备状态。

## 5. 必须实现的功能需求

## 5.1 用户与老人档案管理

手机端应建立老人个人档案，至少包含：

```text
用户ID
老人姓名
年龄段
居住房间
紧急联系人
日常起床时间
日常睡眠时间
正常活动时段
正常步数范围
常用设备
常用房间
常用机器人任务
提醒偏好
告警通知方式
```

推荐数据结构：

```json
{
  "elder_id": "elder_001",
  "name": "用户A",
  "age_group": "70-79",
  "home_id": "home_001",
  "default_room": "bedroom",
  "routine": {
    "wake_time": "08:00",
    "sleep_time": "22:30",
    "normal_steps_min": 500,
    "normal_steps_max": 4000
  },
  "preferences": {
    "voice_volume": 70,
    "reminder_mode": "voice_and_popup"
  }
}
```

老人档案只用于生活规律和风险辅助判断，不作为医疗诊断依据。

---

## 5.2 老人状态数据采集

手机端可采集：

```text
每日签到状态
最后一次手机操作时间
每日步数
应用活跃情况
手机是否在线
手机电量
手动状态确认
主动求助
告警处理结果
家属反馈
```

必须区分：

```text
手机状态异常
老人可能异常
```

例如：

- 手机没电不等于老人异常；
- 应用被系统关闭不等于老人异常；
- 步数低不等于跌倒；
- 未签到只能作为低置信度参考。

统一采集格式建议：

```json
{
  "source": "mobile",
  "elder_id": "elder_001",
  "timestamp": 1783824000,
  "data": {
    "checked_in": true,
    "steps_today": 1850,
    "last_phone_active": 1783823600,
    "manual_status": "normal",
    "phone_online": true,
    "phone_battery": 62
  }
}
```

## 5.3 老人状态辅助判断

手机端不直接给出医学结论，而是生成辅助状态：

```text
NORMAL              正常
NOTICE              需要关注
PENDING_CONFIRM     等待确认
HIGH_RISK           高风险
DEVICE_FAULT        设备故障
UNKNOWN             数据不足
```

推荐判断逻辑：

```text
长时间未签到
    -> NOTICE

手机活动异常下降
    -> NOTICE

手机异常 + RDK疑似跌倒
    -> PENDING_CONFIRM

RDK疑似跌倒 + 语音无应答
    -> HIGH_RISK

ESP32离线或RDK离线
    -> DEVICE_FAULT
```

个人模型应基于老人自己的长期基线，而不是使用统一固定阈值。

## 5.4 ESP32 终端发现与绑定

手机端应支持：

```text
扫描或发现 ESP32 终端
显示设备 ID
显示固件版本
显示协议版本
首次配对
输入配对码
绑定老人档案
解绑设备
查看最近在线时间
```

基础版建议采用：

```text
手机连接 ESP32 SoftAP
ESP32 固定地址 192.168.4.1
手机通过本地 HTTP 与 ESP32 通信
```

后续可扩展：

```text
BLE 配网
二维码配对
局域网 UDP/mDNS 发现
RDK 辅助配网
```

## 5.5 手机与 ESP32 握手

### 能力查询

```http
GET /api/v1/hello
```

ESP32 返回：

```json
{
  "protocol_version": 1,
  "device_id": "c5_terminal_001",
  "firmware_version": "7.4.0",
  "device_type": "elder_iot_terminal",
  "session_required": true,
  "capabilities": [
    "sensor_status",
    "robot_control",
    "wifi_provision",
    "emergency_stop"
  ]
}
```

### 建立会话

手机发送：

```json
{
  "client_id": "phone_001",
  "protocol_version": 1,
  "nonce": "a81f39d2",
  "pairing_code": "482913"
}
```

ESP32 返回：

```json
{
  "session_id": "ses_1001",
  "accepted": true,
  "heartbeat_interval_ms": 5000,
  "max_payload_bytes": 1024
}
```

### 心跳

手机每 5 秒发送一次：

```json
{
  "session_id": "ses_1001",
  "sequence": 18,
  "timestamp": 1783824000
}
```

连续多个周期无响应后，客户端将 ESP32 标记为离线。

手机端应统一管理：

```text
ESP32终端
RDK小车
智能灯
智能插座
空调
窗帘
其他可扩展设备
```

推荐设备模型：

```json
{
  "device_id": "light_livingroom_01",
  "device_type": "light",
  "name": "客厅主灯",
  "room": "living_room",
  "online": true,
  "capabilities": [
    "set_power",
    "set_brightness"
  ],
  "last_seen": 1783824000,
  "actual_state": {
    "power": true,
    "brightness": 70

}
```

客户端界面根据 `capabilities` 动态生成控制项，避免针对每一种具体设备型号写死页面。

## 5.7 设备控制

手机端应支持：

```text
查询设备状态
打开或关闭设备
设置设备参数
启动小车巡检
让小车前往指定房间
让小车回充
停止当前任务
紧急停止
```

统一命令格式：

```json
{
  "version": 1,
  "request_id": "req_1001",
  "target": "robot_home_01",
  "action": "patrol",
  "params": {},
  "timestamp": 1783824000,
  "requires_confirmation": false
}
```

设备执行结果：

```json
{
  "version": 1,
  "request_id": "req_1001",
  "status": "running",
  "message": "机器人正在巡检",
  "timestamp": 1783824002
}
```

最终结果：

```json
{
  "version": 1,
  "request_id": "req_1001",
  "status": "succeeded",
  "actual_state": {
    "task": "idle",
    "location": "charging_station"
  }
}
```

## 5.8 命令状态跟踪

命令状态必须包括：

```text
PENDING
SENT
ACCEPTED
RUNNING
SUCCEEDED
FAILED
CANCELLED
TIMEOUT
DEVICE_OFFLINE
```

手机端不能在“HTTP发送成功”后立即显示设备已经执行。

必须记录：

```text
request_id
目标设备
动作
发送时间
当前状态
最后更新时间
错误码
失败原因
```

## 5.9 RDK 小车控制与状态

手机端只发送高层任务：

```text
goto
patrol
return_home
inspect
stop
emergency_stop
query_status
```

不直接发送：

```text
PWM
轮速
底盘电机参数

```

手机端显示：

```text
小车在线状态
当前位置
当前任务
任务进度
电量
充电状态
是否受阻
最近错误
视觉异常事件
```

## 5.10 智能家居控制

基础版可优先支持：

```text
智能灯
智能插座
模拟空调
模拟窗帘
```

建议先采用统一虚拟设备模型完成比赛闭环，再逐步接入真实设备协议。

控制流程：

```text
手机发送命令
  -> ESP32或RDK校验
  -> 设备执行
  -> 返回ACK
  -> 返回实际状态
  -> 手机更新显示
```

不能只显示“已发送”，必须显示“实际已执行”或“执行失败”。

## 5.14 本地离线能力

外网断开时，手机端仍应支持：

```text
连接ESP32 SoftAP
查询本地状态
控制基础设备
发送紧急停止
查看RDK在线状态
查看当前传感器数据

```

不应要求所有功能必须依赖云端。

## 5.15 网络恢复和数据补传

手机端应缓存：

```text
老人状态采集数据
签到记录
告警处理结果
少量未上传事件
设备控制历史
```

恢复网络后再补传。

对于控制命令：

- 不可无限重发；
- 必须通过 `request_id` 去重；
- 非幂等命令超时后需要人工确认；
- 紧急停止可重复发送。

## 6. 推荐的软件通信方案

## 6.1 手机与ESP32

基础版推荐：

```text
SoftAP + HTTP + JSON
```

优点：

```text
Android开发简单
便于抓包
无需部署MQTT客户端
适合固定本地地址
调试成本低
```

限制：

```text
手机需要连接ESP32热点
可能暂时失去原Wi-Fi网络
不适合大量实时数据
```

后续可增加：

```text
BLE 配网
局域网 HTTP
mDNS发现
WebSocket状态推送
```

---

## 6.2 ESP32与RDK

推荐：

```text
USB串口作为主控制链路
局域网TCP或MQTT作为状态和扩展链路
```

手机端不需要理解ESP32与RDK底层帧格式，只读取统一状态。

---

## 6.3 RDK与云端

推荐由RDK承担：

```text
MQTT/TLS
HTTPS
ASR
LLM
TTS
历史数据上传
复杂JSON解析
```

这样可以避免ESP32承担过多TLS、音频和大JSON压力。


## 8. 安全要求

手机端至少需要：

```text
首次配对码
设备绑定
会话ID
请求序号
时间戳
命令去重
控制权限
加密存储
敏感日志脱敏
解绑和恢复出厂机制
```

高风险操作需要二次确认：

```text
门锁
大功率插座
远程喊话
小车移动
紧急联系人通知
恢复出厂设置
```

紧急停止除外，应始终可快速触发。

## 10. 性能与资源要求

### 手机端

建议：

```text
后台任务不高频轮询
优先使用事件和系统调度
避免持续保持摄像头和麦克风
单条控制消息小于2KB
状态列表分页加载
日志限制大小
```

### ESP32对接限制

手机端应遵守：

```text
普通请求体建议小于1KB
单次响应建议小于2KB
不要向ESP32上传大图片
不要向ESP32发送长音频
不要要求ESP32保存完整历史数据
不要频繁刷新全部设备列表
```

建议刷新周期：

```text
ESP32连接心跳：5秒
RDK状态：1至2秒
设备状态：状态变化时推送或2至5秒
老人基础状态：分钟级
环境数据：约10秒
```

## 16. 最终结论

手机终端应被建设为：

> **老人状态辅助判断入口、家庭物联网设备总控入口、机器人任务入口、告警处理入口和多端状态汇总入口。**

最可行的开发方向是：

```text
先实现可靠连接
再实现状态同步
再实现命令闭环
再实现老人模型
最后加入AI和自动调度
```

项目开发中必须遵循：

```text
手机负责用户交互和数据汇总
ESP32负责本地终端和安全仲裁
RDK负责运动、视觉和复杂计算
智能家居设备负责实际执行
云端负责长期数据和远程服务
```

只有先完成设备身份、握手、心跳、命令ACK和实际状态回传，才能形成可靠、可展示、可扩展的物联网比赛项目。
