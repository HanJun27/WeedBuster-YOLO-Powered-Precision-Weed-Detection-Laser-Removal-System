## 1. 引言

### 1.1 编写目的

本文档根据现有实物硬件重新限定 ESP32-C5 与 RDK X5 的职责、可连接外设数量、通信方式和验收边界，避免提出当前硬件无法同时满足的要求。

本版本遵循以下实际条件：

1. ESP-SensairShuttle 的外接传感器接口一次只安装并使用一个传感器模块。
2. BME690、BMI270、BMM350 不作为同时在线的并行传感器组合进行验收。
3. LCD、触摸、麦克风和扬声器属于终端固定功能，不计入外接传感器模块数量。
4. RDK X5 小车已经具备自身的底盘运动、姿态、导航和避障相关模块，ESP32-C5 不直接驱动小车电机。
5. ESP32-C5 主要承担用户交互、单模块传感器管理、联网、AI 链路验证、安全确认和备用网关功能。
6. RDK X5 主要承担视觉计算、机器人运动控制、导航以及主通信中继。

### 1.2 系统硬件架构

```text
┌───────────────────────────────────────────────────────────────┐
│                    云端服务 / 边缘大模型网关                    │
│    后端服务、消息存储、告警通知、ASR/LLM/TTS、设备管理          │
└───────────────────────────┬───────────────────────────────────┘
                            │ HTTPS / MQTT（由 RDK 主承载）
┌───────────────────────────┴───────────────────────────────────┐
│                    RDK X5 小车（主计算与执行）                   │
│  视觉检测 │ 导航避障 │ 底盘运动 │ 状态管理 │ 网关代理/云通信     │
└───────────────────────────┬───────────────────────────────────┘
                            │ USB 串行主链路
                            │ 局域网链路作为扩展/备用
┌───────────────────────────┴───────────────────────────────────┐
│                  ESP32-C5（用户终端与安全前端）                  │
│  LCD触控 │ Wi-Fi管理 │ 音频采集/播放 │ AI核对 │ 安全仲裁          │
│  单外接传感器管理 │ 天气链路验证 │ RDK状态显示 │ 备用直连网关       │
└───────────────┬───────────────────────────────────────────────┘
                │ 共享 I²C 扩展接口：一次仅安装一个传感器模块
        ┌───────┴──────────────────────────────────┐
        │ BME690 或 BMI270 或 BMM350（三选一）       │
        └──────────────────────────────────────────┘
```

### 1.3 节点职责

| 节点 | 硬件平台 | 必须承担的职责 | 不承担的职责 |
|---|---|---|---|
| ESP32-C5 终端 | ESP-SensairShuttle | LCD触控、单传感器模块识别与采集、Wi-Fi、密码键盘、音频接口、天气链路验证、AI结果核对、安全确认、RDK通信、备用网关 | 不直接驱动底盘电机；不要求同时读取三个外接传感器；不承担主要视觉计算 |
| RDK X5 小车 | RDK X5 ROS 小车 | 摄像头/视觉、导航、避障、底盘运动、机器人状态、主要边缘计算、主要云端/网关代理 | 不负责 ESP32 屏幕和外接传感器底层驱动 |
| 云端/网关 | 后端与大模型服务 | 消息存储、远程通知、模型推理、ASR/LLM/TTS、设备管理 | 不直接绕过本地安全规则控制底盘 |
| 外部智能灯 | 可选扩展节点 | 接收本地告警联动命令 | 不列入当前基础版必达硬件 |

## 2. 硬件接口规范

### 2.1 ESP32-C5 固定接口

| 接口 | 外设 | 当前约束 | 说明 |
|---|---|---|---|
| SPI LCD | ST7789 系列 LCD | 240×284 | 显示安卓图标式主页、状态、键盘和诊断信息 |
| I²C | CST816T/CST816S 触摸 | 地址 0x15 | 固定触摸设备，与传感器总线共享时必须由统一总线管理器维护 |
| ADC | 模拟麦克风 | GPIO6 / ADC1_CH5 | 16kHz 单声道录音基础链路 |
| PDM TX | 扬声器输出 | GPIO7 / GPIO8 | 播放提示音和短语音数据 |
| Wi-Fi | 2.4GHz WLAN | 自动联网、保存凭据 | 所有联网业务必须先检查 Wi-Fi 管理器状态 |
| USB Serial/JTAG | RDK X5 主链路 | 带帧校验 | 传输心跳、状态、AI代理请求和机器人高层命令 |

### 2.2 单外接传感器接口

外接传感器扩展接口一次只能安装一个模块。系统启动时必须扫描并识别当前模块，只初始化实际存在的设备。

| 模块模式 | I²C 地址 | 默认速率 | 主要数据 | 使用限制 |
|---|---:|---:|---|---|
| BME690 环境模式 | 0x76 / 0x77 | 100kHz | 温度、湿度、气压、气体电阻 | 不与 BMI270、BMM350 同时作为外接模块使用 |
| BMI270 惯性模式 | 0x68 / 0x69 | 100kHz | 加速度、角速度 | 仅用于终端姿态/调试，不作为小车运动模块替代品 |
| BMM350 磁力模式 | 0x14 | 100kHz | 三轴磁场 | 仅用于方向/调试，不作为基础版必需数据源 |

要求：

1. 上电后先锁定 CS、SDO 等模式脚，再扫描地址。
2. 支持识别 `ENVIRONMENT`、`IMU`、`MAGNETOMETER` 和 `NONE` 四种板卡状态。
3. 不允许同时创建三个传感器的有效设备句柄。
4. 传感器更换操作要求先断电；运行期热插拔不列入基础版支持范围。
5. 软件保留重新扫描和 I²C 总线恢复能力，但不得向用户承诺带电插拔安全。

### 2.3 通信协议矩阵

| 链路 | 基础版协议 | Payload | 角色 | 延迟目标 |
|---|---|---|---|---:|
| ESP32-C5 ↔ RDK X5 | USB串行帧，`$payload*CS` | 文本字段或紧凑JSON | 主控制与状态链路 | ≤200ms |
| ESP32-C5 → 大模型 | RDK代理 HTTPS 为主 | JSON | AI主通道 | 由网络条件决定 |
| ESP32-C5 → 大模型 | ESP32直连 HTTPS 为备 | JSON | AI备用通道 | 由网络条件决定 |
| RDK X5 ↔ 云端 | MQTT/TLS 或 HTTPS | JSON | 云通信主通道 | 告警消息≤500ms（网络正常时） |
| ESP32-C5 ↔ 云端 | 预留/进阶 | JSON | 非基础版强制项 | — |
| ESP32-C5 → 外部灯 | ESP-NOW，预留 | 带序号和校验的短帧 | 进阶本地联动 | ≤100ms |

## 3. 功能需求

### HR-1 单传感器模块管理

| 编号 | 需求 | 优先级 |
|---|---|---|
| HR-1.1 | 每次启动扫描 0x14、0x68/0x69、0x76/0x77，并确定唯一的活动外接模块 | M |
| HR-1.2 | 只初始化检测成功的模块；未检测到模块时进入 `NONE` 状态，主系统仍可启动 | M |
| HR-1.3 | BME690 模式下每10秒采集温湿度、气压和气体电阻 | M |
| HR-1.4 | BMI270 模式下以50~100ms周期采集加速度和角速度；中断/FIFO作为后续增强 | S |
| HR-1.5 | BMM350 模式下以100~500ms周期采集三轴磁场数据 | S |
| HR-1.6 | 单次读取失败重试3次；连续5次总线超时后执行 I²C bus recovery 并重建设备句柄 | M |
| HR-1.7 | 恢复失败时将模块标记为 OFFLINE，在 UI 显示地址、chip_id、错误码和重试次数 | M |
| HR-1.8 | 更换模块必须断电；重新上电后系统自动识别新模块，不依赖人工修改固件 | M |

统一状态结构：

```json
{
  "active_module": "BME690",
  "address": "0x76",
  "online": true,
  "chip_id": "0x61",
  "retry_count": 0,
  "last_error": "NONE"
}
```

BME690 基础遥测格式：

```json
{
  "ts": 1712345678,
  "module": "BME690",
  "temperature_c": 25.3,
  "humidity_percent": 58.2,
  "pressure_pa": 101325,
  "gas_resistance_ohm": 120000
}
```

说明：未接入正式 IAQ 算法前，不将气体电阻伪装成标准 IAQ 数值。

### HR-2 RDK X5 视觉与运动控制

| 编号 | 需求 | 优先级 |
|---|---|---|
| HR-2.1 | RDK负责底盘电机、编码器、导航、避障、摄像头和运动控制模块 | M |
| HR-2.2 | ESP32只发送高层任务命令：`goto`、`patrol`、`stop`、`emergency_stop`、`follow_person` | M |
| HR-2.3 | RDK返回 `accepted`、`running`、`arrived/succeeded`、`blocked`、`failed` 状态 | M |
| HR-2.4 | 跌倒识别以 RDK 摄像头/视觉模型为主要触发源 | M |
| HR-2.5 | ESP32上的 BMI270 仅在安装该模块时提供辅助数据，不能作为基础版固定跌倒数据源 | M |
| HR-2.6 | RDK失联时 ESP32禁止继续发送运动命令，并保留紧急停止入口 | M |

机器人任务示例：

```json
{
  "cmd_id": "cmd_4e5f6a7b",
  "cmd": "goto",
  "params": {
    "target": "bedroom"
  }
}
```

### HR-3 本地告警与安全状态机

ESP32-C5保留确定性的安全确认状态机，但其主要事件来源为 RDK 视觉结果，而不是多个外接传感器融合。

| 状态 | 触发条件 | 动作 | 超时 |
|---|---|---|---|
| IDLE | 初始状态或事件结束 | 显示正常状态，监听RDK事件 | — |
| PENDING | RDK发送跌倒疑似事件且置信度达到阈值 | 显示核对页、播放“您还好吗”、启动应答计时 | 10s |
| LOG_ONLY | 用户明确表示无事 | 记录事件，不升级告警，返回IDLE | — |
| HIGH_ALERT | 无应答、明确求助或人工确认告警 | 本地报警音、通知RDK/云端、显示高危页面 | — |

安全要求：

1. AI自然语言输出不得直接控制小车。
2. 所有运动命令必须转换为固定协议，并经过目标白名单和参数检查。
3. `stop` 和 `emergency_stop` 不依赖大模型判断。
4. 天气测试和AI自检不得产生任何运动命令。
5. RDK离线、协议异常或AI自检未通过时，拒绝进入自动执行状态。

|---|---|---|

| HR-5.3 | 软键盘当前只用于 Wi-Fi 密码，支持大小写英文字母和数字 | M |

### HR-6 AI链路与天气验证

| 编号 | 需求 | 优先级 |
|---|---|---|
| HR-6.1 | AI模块启动前必须先执行链路自检 | M |
| HR-6.2 | 自检固定使用中国地区天气类只读问题，不调用实际硬件动作 | M |


### HR-7 音频与语音交互

| 编号 | 需求 | 优先级 |
|---|---|---|
| HR-7.1 | 使用 ADC 采集16kHz、16bit、单声道录音，支持静音自动停止 | M |
| HR-7.2 | 使用 PDM 输出播放提示音和短音频 | M |
| HR-7.3 | 完整语音链路采用 ASR -> LLM -> TTS，不向仅支持文本的模型直接发送音频 | M |

### HR-8 云端与消息通信

基础版不要求 ESP32-C5 同时维持云端 EMQX 和 RDK 本地 Mosquitto 两个 MQTT 客户端。云端通信优先由 RDK X5 承担，以减少 ESP32 内存和网络复杂度。

| 编号 | 需求 | 优先级 |
|---|---|---|
| HR-8.1 | ESP32将传感器状态、告警状态和用户确认结果发送给RDK | M |
| HR-8.2 | RDK负责将遥测、心跳、事件和命令状态上报云端 | M |
| HR-8.3 | RDK接收云端命令后先做权限和格式校验，再转发高层任务给ESP32或底盘模块 | M |
| HR-8.4 | ESP32直连MQTT/TLS作为后续备用通道，不列为当前三周基础版必达 | S |
| HR-8.5 | 云端消息必须带 `event_id/cmd_id`，支持幂等和状态回显 | M |

建议云端 Topic：

| Topic | 主要发布方 | 说明 |
|---|---|---|
| `home/{id}/telemetry` | RDK | 环境遥测和设备状态 |
| `home/{id}/event` | RDK | 跌倒/告警事件 |
| `home/{id}/heartbeat` | RDK | C5与RDK汇总心跳 |
| `home/{id}/cmd` | 云端 | 远程高层任务 |
| `home/{id}/cmd/ack` | RDK | 指令状态回显 |
| `robot/{id}/event/vision` | RDK内部/云端 | 视觉事件 |

## 4. 非功能需求

| 编号 | 类别 | 指标 | 验证方式 |
|---|---|---|---|
| HNFR-1 | 单模块约束 | 任意时刻只允许一个外接传感器模块处于ACTIVE状态 | 日志与UI检查 |

| HNFR-5 | RDK实时性 | RDK事件到ESP32接收≤200ms | 两端时间戳 |

| HNFR-8 | Wi-Fi韧性 | 已保存网络连接失败后给出明确提示，且不阻塞离线功能 | 关闭AP测试 |
| HNFR-9 | AI安全 | 自检失败、协议不合法或RDK离线时不得执行自动运动命令 | 故障注入 |
| HNFR-10 | 资源 | ESP32稳定运行时空闲堆内存≥40KB | 堆内存日志 |
| HNFR-11 | 适老化 | 核心状态和告警使用高对比大字；辅助说明允许较小字号 | 目测与页面评审 |
| HNFR-12 | 存储安全 | Wi-Fi凭据不得输出到普通日志；正式部署启用NVS/Flash安全策略 | 日志检查与配置检查 |

## 5. 版本范围

### 5.1 当前基础版必达

1. ESP32启动时识别一个外接传感器模块，并只初始化该模块。
2. 支持 BME690、BMI270、BMM350 三种单模块模式切换。
3. 修复断电更换模块后偶发无法读取地址/chip_id的问题。
4. 安卓图标式主页、手势翻页、页码显示和右滑返回。
5. 系统Wi-Fi模块、密码键盘、凭据记忆和自动联网。
6. 天气链路独立验证页面和AI启动自检。
7. RDK心跳、状态接收、STOP和EMERGENCY_STOP。
8. RDK处理底盘运动、导航和视觉，ESP32不直接控制电机。
9. AI返回固定协议、人工核对和安全白名单。
10. 网络不可用时保留UI、传感器、告警音和紧急停止。

### 5.2 进阶版

1. 离线唤醒词和可打断的流式语音交互。
2. ESP32直连MQTT/TLS备用通道。
3. ESP-NOW智能灯或其他本地联动节点。
4. 多传感器扩展坞；只有增加独立硬件接口后才要求并行采集。
5. 运行期安全热插拔；需增加电源开关、缓冲和防倒灌设计后再立项。
6. BMI270 FIFO/中断高级算法。
7. BME690正式IAQ算法。
8. RDK人体跟随和更复杂的自主巡检。

## 6. ESP32-C5 与 RDK 协议冻结建议

### 6.1 帧格式

```text
$<payload>*<XOR_CHECKSUM>\r\n
```

### 6.2 基础消息

| 方向 | Payload 示例 | 说明 |
|---|---|---|
| RDK→C5 | `HEARTBEAT,STATE=READY` | RDK在线心跳 |
| RDK→C5 | `ROBOT_STATUS,STATE=RUNNING,TARGET=BEDROOM` | 机器人状态 |
| RDK→C5 | `FALL_EVENT,ID=evt01,CONF=0.86,ROOM=BEDROOM` | 跌倒疑似事件 |
| C5→RDK | `CMD,STOP` | 普通停止 |
| C5→RDK | `CMD,EMERGENCY_STOP` | 紧急停止 |
| C5→RDK | `CMD,GOTO,TARGET=BEDROOM,ID=cmd01` | 前往目标点 |
| C5→RDK | `CMD,PATROL,ID=cmd02` | 巡检 |
| C5→RDK | `AI_REQ,ID=req01,TEXT=...` | AI代理请求 |
| RDK→C5 | `AI_RESP,ID=req01,VALID=1,TEXT=...` | AI代理结果 |

### 6.3 单传感器状态上报

```json
{
  "type": "sensor_status",
  "active_module": "BME690",
  "online": true,
  "address": 118,
  "last_error": 0
}
```

## 7. 验收标准

### 7.1 ESP32单元验收

| # | 验收项 | 方法 | 通过标准 |
|---|---|---|---|
| T-1 | 单模块识别 | 分别安装BME690、BMI270、BMM350后断电启动 | 自动识别正确模块和地址，只初始化一个模块 |
| T-2 | 模块轮换 | 断电更换模块并重新上电 | 不需改固件即可识别；失败时给出明确错误而非随机卡死 |
| T-3 | I²C恢复 | 人为拉低SDA/制造超时 | 触发恢复并允许系统继续进入UI |
| T-4 | UI分页 | 左右滑动主页 | 正确翻页、显示页码、点击不误触发滑动 |
| T-5 | Wi-Fi | 使用已保存网络和错误密码分别测试 | 自动联网成功；失败时显示检查网络/密码 |
| T-6 | 天气链路 | 启动真实验证和本地样例 | 返回协议状态；不产生任何硬件动作 |
| T-7 | AI阻断 | 断网或返回非法JSON | AI执行被阻止，并显示失败阶段 |
| T-8 | 音频 | 录音、静音停止、固定提示音播放 | 功能可重复使用且不阻塞UI |

### 7.2 C5与RDK集成验收

| # | 验收项 | 方法 | 通过标准 |
|---|---|---|---|
| T-9 | RDK心跳 | 启动小车与ESP32 | ESP32显示RDK在线，超时后显示离线 |
| T-10 | 高层运动命令 | 发送goto/patrol/stop | RDK执行并返回状态；ESP32不直接输出电机控制信号 |
| T-11 | 紧急停止 | 小车运动中触发 | 紧急命令优先发送，RDK停止并回传状态 |
| T-12 | 跌倒事件 | RDK发送FALL_EVENT | ESP32进入PENDING页面并启动语音/触控核对 |
| T-13 | 完整告警 | 模拟无应答或求助 | 进入HIGH_ALERT，播放本地告警并通知RDK/云端代理 |

### 7.3 明确不作为当前基础版验收项

1. 三个外接传感器同时在线。
2. ESP32直接控制小车电机、编码器或底盘驱动。
3. ESP32同时维持云端和本地两个MQTT客户端。
4. 运行期带电更换传感器模块。
5. ESP-NOW智能灯、毫米波哨兵节点和离线唤醒词。
6. 未接入算法时的标准IAQ值。

| 任务 | FreeRTOS优先级 | 周期/触发 | 说明 |
|---|---:|---|---|
| RDK接收与紧急命令 | 7 | 事件驱动 | 紧急停止优先级最高 |

| Wi-Fi与网关路由 | 4~5 | 事件驱动/重连 | 所有联网业务统一入口 |
| LCD刷新与触摸 | 3~4 | 40~250ms | 使用局部刷新和快照比对 |
| 日志与诊断 | 2 | 低频 | 不阻塞控制链路 |

## 9. 风险与应对

| 风险 | 影响 | 应对 |
|---|---|---|
| 误把单接口当作多传感器并行总线 | 无法满足数据融合需求 | 文档明确三选一；融合改由RDK视觉和当前活动模块完成 |
| 断电换模块后地址偶发不可读 | 模块启动失败 | 模式脚硬件上下拉、上电延时、地址双探测、chip_id重试、bus recovery |
| 用户带电插拔 | 总线锁死或反向供电 | UI和文档提示必须断电更换；基础版不承诺热插拔 |
| ESP32承担过多云通信 | 内存不足、链路复杂 | RDK承担主MQTT和网关代理，ESP32保留备用HTTPS |
| 大模型直接输出底盘动作 | 安全风险 | 固定协议、白名单、人工确认、紧急停止独立通道 |
| RDK失联 | 无法运动或视觉复核 | 禁止新运动命令，保留本地告警和离线UI |
| BME690无正式IAQ算法 | 数据含义不准确 | 基础版仅报告气体电阻，IAQ列为进阶项 |
| 屏幕空间有限 | 页面拥挤 | 安卓图标、手势分页、页码、分级页面和局部刷新 |

## 10. 需求冻结结论

当前硬件版本的核心边界如下：

```text
ESP32-C5 = 用户终端 + 单传感器前端 + 联网/AI安全入口 + RDK备用网关
RDK X5   = 视觉计算 + 导航避障 + 底盘运动 + 主网关/云通信
传感器   = BME690 / BMI270 / BMM350 一次只使用一个
运动模块 = 全部由小车已有硬件和RDK软件负责
```

后续若增加多传感器扩展板、独立电源开关或更多总线接口，应另行发布硬件需求 v2.0，不在本版本中提前要求并行采集或安全热插拔。
